# ADR-002: Structured Output Strategy Per Backend

**Status**: Proposed
**Date**: 2026-04-18

## Context

Each backend returns a `SessionIntent` JSON object, but the mechanism for ensuring the JSON is valid and schema-compliant differs fundamentally across backends:

**Claude CLI subprocess**: The CLI's `--output-format json` flag returns a wrapper struct (metadata envelope) — the actual model response is nested inside a `result` field. The inner content is plain text generated by the model; it is not constrained to any schema. The model is trained to wrap code/JSON in markdown fences, and frequently adds explanatory prose before or after JSON objects.

**Anthropic SDK**: The SDK exposes `output_config.format` with a JSON schema parameter. This triggers constrained decoding (grammar compiled from the schema), which guarantees the response body is valid JSON matching the schema exactly. The overhead is 100–300ms for first-use grammar compilation; the compiled grammar is cached 24 hours. This is GA for Claude Sonnet 4.5+, Haiku 4.5+, Opus 4.5+.

**AWS Bedrock**: The Converse API exposes `outputConfig.textFormat.type: "json_schema"` with a `structure.jsonSchema.schema` field. Same constrained-decoding mechanism as the Anthropic API, GA for Claude 4.5 models across all commercial Bedrock regions.

The research identified that 70% of CLI subprocess responses include a markdown code-block wrapper around the JSON, and 65% include explanatory preamble or postamble text — both of which cause `json.Unmarshal` to fail if applied directly to the raw output. The SDK and Bedrock backends do not have this problem when structured outputs mode is used.

The pitfalls research also confirmed a tail-latency risk for the CLI backend: network-dependent startup (the Grove notice config fetch) can push first-invocation time to 10–12s. The synthesis set a 10s hard timeout (not the 3s listed in the original requirements spec) to accommodate this.

## Decision

Each backend uses a different extraction strategy, all hidden behind the `ParseIntent` method:

**CLIBackend (prompt-based + extraction + repair)**:
1. Invoke `claude --output-format json -p "<system_prompt>\n\nUser: <description>"` with a 10-second `context.WithTimeout`
2. Extract the `result` field from the outer CLI wrapper JSON
3. Run a JSON extraction pass on the inner string: attempt `json.Unmarshal` directly; if that fails, strip markdown fences (```` ```json ... ``` ````), then find the first `{...}` object (brace-depth counting, not regex)
4. Validate the extracted object against the `SessionIntent` fields; default missing optional fields; reject if `Path` is empty
5. Validate `SuggestedSessionID` against the session store; zero it out if the session doesn't exist

**AnthropicSDKBackend (constrained decoding)**:
1. Define the `SessionIntent` JSON schema and pass it via `output_config.format`
2. Force structured output: the model response body is schema-guaranteed valid JSON
3. Inject top-10 most-recent sessions as text into the system prompt for context (no tool calls in v1; MCP tool integration is a v1.1 enhancement)
4. Timeout: 15 seconds (SDK adds network latency; constrained decoding is slower on first call per schema)
5. Validate `SuggestedSessionID` against the session store

**BedrockBackend (constrained decoding via Converse API)**:
1. Use `bedrockruntime.ConverseInput` with `OutputConfig.TextFormat` set to `json_schema` containing the `SessionIntent` schema
2. Same context injection pattern as SDK backend (top-10 sessions in system prompt)
3. Use AWS SDK v2 credential chain (instance profile, environment, shared credentials file) — no credential management in backend code
4. Timeout: 20 seconds (Bedrock adds ~500ms routing overhead; credential chain resolution can add 100–300ms)
5. Validate `SuggestedSessionID` against the session store

**Shared pre-warm**: On server startup, before the first user request, invoke `claude --version` as a subprocess. This pre-loads the Node.js runtime and reduces first-invocation cold-start latency for the CLI backend.

**Shared JSON repair utility** (`server/intent/json_extract.go`):
- `ExtractJSON(raw string) (string, error)` — handles markdown fences, leading/trailing prose, and nested objects
- Used only by `CLIBackend`; SDK and Bedrock backends receive schema-compliant JSON directly

## Consequences

**Positive**:
- SDK and Bedrock backends have schema-compliance guarantees; JSON parse errors are not a runtime failure mode for those paths
- CLI backend's extraction logic is centralized in one file and unit-testable with a table of LLM output fixtures
- The 10s CLI timeout accommodates real-world tail latency (Grove notice config fetch); the original 3s target would have caused spurious timeouts

**Negative / accepted costs**:
- `CLIBackend` cannot guarantee schema compliance; field validation must catch invalid values (e.g., `Program` not in `["claude", "aider"]`, branch names containing spaces)
- The 100–300ms schema compilation overhead on first SDK/Bedrock call adds latency that is not amortized on the first user request after server restart (subsequent calls use the cached grammar)
- `output_config.format` is not compatible with message prefilling, ruling out field-by-field streaming pre-fill (a v1.1 enhancement) for the SDK backend in its current form

**Rejected alternatives**:
- *Prompt-only JSON for all backends*: Too fragile for SDK and Bedrock — the constrained decoding is available, tested, and eliminates the highest-likelihood failure modes (2.1, 2.2 from the pitfalls list)
- *Tool-use / function-calling for structured output*: Valid alternative; the `output_config.format` approach is simpler for v1 since it does not require a tool-call loop and is fully supported by both the Go SDK and Bedrock Converse
- *Regex-based JSON extraction as primary strategy*: Rejected; brace-depth counting (stack-based) is more correct than regex for nested JSON objects
